## рҹ”¬ What Makes Us Different

### Intensity Г— Duration вҖ” Not Just Rainfall

Most parametric insurance systems look at a single number: total rainfall in mm. We looked deeper.

> **Other teams:** `Total Rainfall (mm)` вҶ’ trigger
> 
> **SaaralCare:** `Rainfall Intensity (mm/hr) Г— Duration (hrs)` вҶ’ trigger

This distinction matters enormously. A 10mm drizzle spread over 5 hours is very different from a 10mm downpour in 20 minutes. Our model captures *both* dimensions.

---

### Peak-Hour Intelligence вҖ” The Secret Weapon

Here's what no one else thought about: **not all rain is equally damaging.**

Rain at 3 AM doesn't hurt a delivery worker's income. Rain at 1 PM during the lunch rush absolutely destroys it.

We identified **zone-specific peak delivery hours** through rigorous research:

- рҹҚҪпёҸ **Lunch peak:** 1:00 PM вҖ“ 3:00 PM
- рҹҢҷ **Dinner peak:** 8:00 PM вҖ“ 10:00 PM *(varies by zone)*

We systematically sampled restaurants across Chennai's zones, identified true peak ordering windows, and built those patterns into our trigger engine. The result: our system weights rainfall disruption based on *when* it actually hurts workers' income.

```
вҳ” Rainfall during PEAK hours:
   Trigger = 1.5 Г— Intensity (mm/hr) Г— Duration (hrs)

рҹҢӮ Rainfall during NON-PEAK hours:
   Trigger = Intensity (mm/hr) Г— Duration (hrs)
```

The `1.5Г—` multiplier reflects a fundamental truth: **losing peak-hour earnings is 50% more damaging than losing off-peak earnings.** Our model respects that.

---

### Multi-Point Spatial Averaging

For each zone, we don't just take one weather reading. We query Open-Meteo at:
- The zone center
- 1 km North, South, East, West

Then we **average all five readings** to get a robust, spoof-resistant rainfall estimate for that zone вҖ” every single hour.

## рҹ”’ Advanced Fraud Prevention Architecture

> _SaaralCare doesn't detect fraud. It makes fraud architecturally impossible._

This is our core philosophy: **remove every entry point for fraud instead of trying to catch it after the fact.**

### 1 вҖ” No-Claim System
Workers **cannot**:
- Submit claims
- Upload evidence
- Request payouts manually

вҶ’ **Eliminates:** Fake claims, inflated loss reporting, manual manipulation

### 2 вҖ” Immutable External Data Source
Payouts depend *only* on **Open-Meteo API** data вҖ” an external, trusted source with zero user influence.

вҶ’ **Eliminates:** Data tampering, internal manipulation of triggers

### 3 вҖ” No GPS (Kills Location Spoofing)
We made a deliberate architectural choice: **no mobile GPS, ever**.

Other systems that rely on GPS location tracking open themselves to spoofing apps, VPN-based location fraud, and geolocation manipulation. We removed the attack surface entirely.

вҶ’ **Eliminates:** GPS spoofing, VPN-based location fraud

### 4 вҖ” Fixed Worker вҶ’ Zone вҶ’ Station Mapping
Every worker is permanently assigned to a **home zone** (fetched from the platform API during onboarding). Rainfall is evaluated using **only that zone's data**.

Workers cannot:
- Change assigned zone
- Select favorable locations
- Switch to high-rainfall zones before storms

вҶ’ **Eliminates:** Location switching fraud, cherry-picking high-rainfall zones

### 5 вҖ” Satellite Cross-Validation (Advanced Fraud Validation)

> _"We don't trust a single data source. We validate against the sky itself."_

Our fraud validation layer cross-references ground station data against **satellite rainfall grids**:

| Layer | Source |
|-------|--------|
| Ground Data | Open-Meteo weather stations |
| Satellite Data | IMERG + TRMM rainfall grids |
| Anomaly Detection | Large mismatches flagged as suspicious |

```
If ground rainfall вү  satellite rainfall вҶ’ System flags anomaly for review
```

**Multi-source validation = Impossible to spoof**

### 6 вҖ” Human-in-the-Loop Admin Review
When the system detects anomalies between ground and satellite data, it surfaces an **anomaly score** on the Admin Portal. Administrators can review flagged events and intervene вҖ” combining algorithmic rigor with human judgment for edge cases.

### 7 вҖ” Policy Activation Delay (T + 7)
Coverage begins **7 days** after premium payment.

вҶ’ **Eliminates:** Buying insurance right before a heavy rain forecast

### 8 вҖ” Platform Worker Verification
Onboarding includes verification against the delivery platform's worker API (mock simulation).

вҶ’ **Eliminates:** Fake identities, duplicate accounts, unauthorized payout access

## рҹ“Ҡ Financial Model

A mathematically rigorous, actuarially-sound parametric pricing engine вҖ” rare at the hackathon level.

### Dual Poisson Risk Framework

Rainfall events are modeled as two **independent Poisson processes**:

| Process | Rate (Base) | Trigger |
|---------|------------|---------|
| Severe rainfall | О»_F = 0.20/week | Full payout |
| Moderate rainfall | О»_P = 0.30/week | Partial payout |
| Total | О»_total = 0.50/week | вҖ” |

### Personalized Risk Adjustment

```
О»_F     = 0.20 Г— s Г— ОІ
О»_P     = 0.30 Г— s Г— ОІ
О»_total = 0.50 Г— s Г— ОІ

Where:
  s вҶ’ Station risk score (from ML model)
  ОІ вҶ’ Seasonal multiplier
```

### Seasonality Multipliers

| Season | ОІ | Effect |
|--------|---|--------|
| Dry | 0.5 | Reduced risk |
| Pre-monsoon | 0.8 | Moderate risk |
| SW Monsoon | 1.2 | Elevated risk |
| NE Monsoon | 1.8 | Peak risk |

### Expected Payout Formula

```
E[Payout] = c_day Г— w Г— E[min(N, 3)]

Where:
  c_day = 0.5 Г— daily income
  w     = 0.70 (weighting factor)
  N     ~ Poisson(О»_total)
  Cap   = max 3 payout days/week

Closed form:
  E[min(N,3)] = 3 вҲ’ e^(вҲ’О»)(3 + 2О» + О»ВІ/2)
```

### Premium Structure

```

P_monthly = 4.33  Г— P_weekly

  12.5% вҶ’ Operations
```

## рҹҡҖ How to Run

### Prerequisites

- Node.js v18+
- Python 3.9+
- Supabase account
- Razorpay account (test mode)

---

### 1 вҖ” Clone the Repository

```bash
git clone https://github.com/Prithvi1247/SaaralCare.git
cd SaaralCare
```

---

### 2 вҖ” Install Dependencies

```bash
npm install
```

---

### 3 вҖ” Configure Environment Variables

Create a `.env.local` file in the root directory:

```env
# в”Җв”Җв”Җ Supabase в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
NEXT_PUBLIC_SUPABASE_URL=your_supabase_url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_supabase_anon_key
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key

# в”Җв”Җв”Җ Razorpay в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
NEXT_PUBLIC_RAZORPAY_KEY_ID=your_razorpay_key_id
RAZORPAY_KEY_SECRET=your_razorpay_secret
```

> вҡ пёҸ **Security Note:** `NEXT_PUBLIC_*` variables are exposed to the browser вҖ” only use public keys here. **Never** expose `RAZORPAY_KEY_SECRET` or `SUPABASE_SERVICE_ROLE_KEY` on the client side.

---

### 4 вҖ” Set Up the Database

Ensure the following tables exist in your Supabase project:

```
вң… workers              вң… zone_station_map
вң… stations             вң… premium_payments
вң… coverage_payout      вң… active_plans
вң… rainfall_events
```

Refer to the schema diagram in the repository for full table definitions.

---

### 5 вҖ” Start the Development Server

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

---

### 6 вҖ” Test the Full Flow

```
1. Login with a demo phone number
2. Complete worker onboarding
3. Purchase a plan (Razorpay test mode)
4. Verify:
   в”ңв”Җв”Җ Payment recorded in premium_payments
   в”ңв”Җв”Җ Plan status updated in workers table
   в””в”Җв”Җ Dashboard reflects active coverage
```

---

### 7 вҖ” Run the ML Weather Risk Model

```bash
# Navigate to model directory
cd Model

# Create and activate a virtual environment (recommended)
python -m venv venv
source venv/bin/activate        # macOS / Linux
# venv\Scripts\activate         # Windows

# Install dependencies
pip install -r requirements.txt

# Run the risk classifier
python weather_risk_predictor.py
```
